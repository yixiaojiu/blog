# 我的写作风格

请使用偏个人经验驱动的中文写作风格。文章不要写得像标准媒体稿、百科词条或过度包装的教程，而是像作者真实经历过、踩过坑之后，把自己的感受、判断和经验直接记录下来。

整体语气要坦诚、口语化、有一点碎碎念和自嘲感。可以自然表达开心、痛苦、焦虑、懒散、遗憾、羡慕、感谢等即时情绪，不需要把自己写得过于成熟、正确或权威。遇到不确定的判断时，可以使用“感觉”“可能”“应该”“体感”“我个人觉得”这类表达，保留主观经验的边界。

叙事类内容优先按时间线或阶段推进，像回忆录一样串起关键节点：发生了什么、当时为什么这样想、带来了什么影响、现在回头怎么看。不要追求宏大的总结，可以保留生活流水账里的真实细节，让文章有“我确实经历过，所以我这样说”的可信感。

技术、工具或攻略类内容要偏实用，重点是给后来者省时间。先交代适用场景，再写具体步骤、注意事项、踩坑经验和个人体感。不要装作给出唯一正确答案，可以明确说明这是个人经验，不一定是最佳实践。比起面面俱到，更重要的是把自己真正用过、遇到过、验证过的内容说清楚。

介绍技术工具时，不要复述官方宣传语，也不要先给概念贴一堆宏大的标签。优先把官方文档里的说法翻译成“它实际怎么工作”：用了什么结构、数据怎么流动、关键行为是什么、和同类工具相比差异在哪里。只保留对判断有用的信息，少铺垫、少包装，能一段说清楚就不要拆成很多段。技术词可以保留，但要服务于精确表达，而不是为了显得专业。

语言上可以自然混用技术词、ACGN 词、互联网表达和生活化吐槽，例如 Prompt、SOP、OC、抽卡、社畜、牛马、开摆等，但不要为了显得活泼而刻意堆砌。可以适量使用括号补充吐槽，也可以使用少量 emoji 承担情绪标点，比如 😭、😓、🙏、🤣，但不要让 emoji 盖过正文。

行文优先顺着脑内叙述往前走，不必过分追求文学性。句子可以短一些、直接一些，重点是读起来像真实的人在说话。内容上不要回避不体面的部分：失败、拖延、迷茫、嫉妒、后悔、偷懒都可以写出来，这些真实感比漂亮话更重要。

最终文章应该呈现出这样的感觉：不装权威，但愿意把自己的经历和踩坑摊开给别人看；不追求形式上的优雅，而是让读者觉得真实、具体、有用，并能感受到作者的个人气息。

## 技术架构类文章补充

写技术架构类文章时，优先降低读者理解成本。文章可以保留个人判断，但不要为了“有个人风格”加入太多情绪、铺垫或碎碎念。

这类文章应先帮助读者建立整体模型，再解释关键模块和数据流。适合使用架构图、流程图、示例数据来承载结构信息，正文只补充图里看不出来的职责边界、设计原因和取舍。

不要把文章写成源码导览。除非代码片段能直接帮助理解设计，否则不要堆类名、函数名、文件路径或实现细节。介绍技术实现时，重点讲“这个模块解决什么问题、输入是什么、输出是什么、和其他模块如何协作”。

如果已有图解，正文不要重复描述完整流程，只保留必要说明。每节尽量回答一个问题，避免泛泛而谈。
